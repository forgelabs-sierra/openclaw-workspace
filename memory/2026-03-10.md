## Key Technical Lessons

### SHA Management in Git-based CMS
- GitHub API requires current blob SHA for updates/deletes
- PUT /api/content returns commitSha (not blob SHA) — must do a fresh GET after save to get new blob SHA
- Same issue after image upload/delete — always refresh SHA after any GitHub write operation
- Consecutive saves were 409ing until this was fixed

### flvercel deploy without --prod = preview deployment
- `flvercel deploy --project my-app --ref feature/cms` creates a preview deploy
- `--prod` flag required for production — don't use it unintentionally
- Accidentally deployed feature/cms to production early in session — fixed by redeploying main

### Branch-pinned preview domains
- `flvercel domains update sierrawins.test.forgelabs.nz --project forgelabs-web --branch feature/cms`
- Pins the custom domain to serve a specific branch's preview deployment
- Very useful for stable preview URLs during feature development

### GITHUB_BRANCH env var pattern
- Set `GITHUB_BRANCH=main` on production target, `GITHUB_BRANCH=feature/cms` on preview target
- All API routes read this var — content and images always read/write to the correct branch
- Essential for multi-branch CMS workflows

### Subagents as coding sandbox
- Spawning subagents for implementation work is effectively a coding sandbox
- Preserves main session context for planning/coordination
- flgh push fails from subagent sandbox (clitools-api unreachable) — main agent must push
- Subagents can build, test (npm run build), commit — just can't push

### Screenshots need time
- Always wait 8-15 seconds after navigate before screenshotting (JS hydration)
- Use browser open → wait → screenshot pattern, not navigate+screenshot in one shot

### Deployment protection
- Vercel deployment protection blocks headless browser (blank screenshots)
- Must be off for public sites — production was fine, preview had it on
- Can't disable via flvercel — must use Vercel dashboard
