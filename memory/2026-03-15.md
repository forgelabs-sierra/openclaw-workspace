- **Parallel task performance:**
  - At concurrency 4: 2.3x more output than sequential
  - Scales further with more concurrent requests
  - Ideal for independent tasks (write code, tests, docs simultaneously)

## Using Parallel Sub-Agents (Mar 15, 16:32)
- **Strategy:** Use `sessions_spawn` for independent tasks rather than sequential execution
- **Example:** Writing code, tests, and docs in parallel instead of one after another
- **Benefit:** Each sub-agent runs concurrently on the DGX, multiplying throughput
- **When to use:** Any task that can be split into independent sub-tasks
- **Avoid for:** Tasks requiring shared state or coordination (use sequential instead)

## Session Audit Tips (Mar 15, 17:46)
- **exec tool host:** Always use `host: "sandbox"` or omit host (defaults to sandbox). Never set `host: "gateway"` — commands run in sandbox container.
- **Memory files:** Use `write` for new daily files, `edit` only for existing files. Don't read first — just write.
- **Read offsets:** Check file length with `wc -l <file>` via exec before using offset to avoid `ENOENT` errors.
- **CLI tools:** `flgog`, `flzotero`, `flvercel`, `flgh`, `fldns` are CLI commands — always run via `exec`, not as standalone tools.
